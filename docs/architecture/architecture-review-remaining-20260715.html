R2. Pull the readiness policy behind a probe seam

Strong ports & adapters policy still fused

Files

Sources/ClaudeCodeRoutesApp/ProxyHealth.swift:25–84

Problem

Extraction moved the UI effect out (good — onStatusChange), but the state policy still lives inside the poll Task, hardcodes URLSession.shared, and depends on real Task.sleep. ProxyHealthChecking has no second adapter. No test can drive starting → not-ready → stopped.

Now · policy inside the task

The probe is not an adapter; policy can't be tested.

Target · pure policy + probe seam

flowchart TB
            OBS[urlUp, processUp, sawReady] --> POL[readiness policy]
            POL --> OUT[healthy + message]
            PR[HTTPProbe seam] --> LIVE[URLSession adapter]
            PR --> FAKE[fake probe in tests]
            classDef deep fill:#0f172a,stroke:#34d399,color:#fff;
            class POL deep;

Policy is a pure function; the probe is the seam with two adapters.

Solution

Extract the decision into a pure function readiness(urlUp:processUp:sawReady:) -> (Bool, String). Put the URL probe behind an HTTPProbe seam with a live URLSession adapter and a fake. Test every transition; the Task just wires probe → policy → callback.

Benefits

  • tests cover all status transitions
  • locality: readiness rules in one function
  • two adapters justify the probe seam
  • leverage: task only wires, never decides
