Deepen command execution

Problem

Two adapters each know how to create, wait for, and drain a Foundation Process; fixes to lifecycle or pipe handling have no locality.

Solution

Move process lifecycle and command outcomes behind one deep command execution module; keep 1Password and provider auth as callers at the seam.

Benefits

  • locality: one process implementation
  • leverage: two callers, one fix
  • tests cross one seam

Deletion test: deleting the second adapter does not make complexity disappear; it reappears as duplicated process code in the remaining callers. That is real depth to consolidate.

Problem

SettingsModel owns persistence, path resolution, async auth state, and callback choreography; AppDelegate owns the restart consequence.

Solution

Concentrate auth action lifecycle, status, and credential-change behavior behind one deep module; callers observe state instead of coordinating callbacks.

Benefits

  • locality: auth rules stay together
  • leverage: one state transition path
  • tests avoid UI choreography

Deletion test: deleting the callbacks would not remove the work; it would scatter auth completion and restart rules across the view and AppDelegate.

Problem

The documented launch contract is ProxySession.apply, but production construction still leaks resolver, runner, planner, and health adapter knowledge into AppDelegate.

Solution

Add a production composition module that owns default adapters while retaining the narrow injected seam used by tests.

Benefits

  • locality: launch wiring has one home
  • leverage: smaller app lifecycle
  • tests keep controlled adapters

Existing design note: README already calls this an intermediate step and suggests a production factory; the friction is real enough to revisit that note.

Problem

Readiness state and user-facing status text live in separate modules; .notReady currently displays “starting…”, so a lost-ready proxy is misreported.

Solution

Let one deep health module own the readiness state, status text, and transition rules; keep process and probe as adapters at its seam.

Benefits

  • locality: semantics and text agree
  • leverage: one status source
  • tests cover transitions once

Concrete defect: fix the misleading “starting…” message even if the larger deepening is deferred.

Concrete defect scan

Safe fixes identified independently of candidate choice

Configuration: blank MERGE_GATEWAY_API_KEY bypasses the 1Password fallback; the proxy path default is hard-coded to one user.

Readiness: .starting is reported healthy before the URL responds, and .notReady is displayed as “starting…”.

Cancellation: a probe result can return after health monitoring is canceled and publish a stale status.

Architecture candidates below: the deeper refactors still need a deliberate seam choice.

Top recommendation

Deepen command execution first

It has the clearest deletion-test result and the highest leverage: two concrete adapters duplicate process lifecycle knowledge today. One deep module would improve locality without changing the proxy launch contract.
